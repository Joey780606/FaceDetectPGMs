## Project Goal

MediaPipe 參考網址: https://ai.google.dev/edge/mediapipe/solutions/vision/face_landmarker

開啟電腦 Webcam，輸入人名後，找出 468 個 3D face landmarks（x, y, z 軸），
萃取 351 維臉部比例特徵向量，依臉部角度,交給相關的類別對 SVM 做分類訓練後，再偵測人臉辨識來者是誰。

## Design Decisions

1. **IOD 歸一化**：所有距離除以瞳距（IOD），消除臉距鏡頭遠近造成的縮放干擾。
2. **Z-Score 標準化**：消除「人類臉部拓撲相似」造成的干擾，使 Cosine 相似度真正具有個人區分能力。
3. **辨識引擎（純 NumPy）**：
   - 多人（≥ 2 人）：OvR 線性 SVM，SGD hinge loss 訓練（`svm_classifier_np.py`）
   - 單人（1 人）：最大 Cosine 相似度比對
   - 不依賴 sklearn / scipy（避免 Windows Application Control 封鎖 DLL）
4. **Unknown 偵測**：不需預先收集陌生人樣本，SVM 分數低於信心度閾值即判為 Unknown。
5. **五類SVM比較**：之前版本會有側臉成效不佳的狀況,此版希望在訓練時,能依照臉部角度變化,分成以下五類:
   正臉,臉朝左上方,臉朝右上方,臉朝左下方,臉朝右下方.
   依MediaPipe取點後,判斷臉的角度進行分類.
   在detect時,依五類評比出最像是誰,並顯示出來.
6. **商用授權**：所有依賴（MediaPipe Apache 2.0、OpenCV Apache 2.0、NumPy BSD、CustomTkinter CC0、Pillow MIT）均為寬鬆授權，可安全商用。

## === 100: 想問問Scores裡面的值,若是負數,就表示不像嗎?
● 是的，完整解釋：

  多人模式（Joey + Golden）

  Score = W @ x + b    ← 線性分類器原始分數

  ┌─────────────┬────────────────────┐
  │  Score 值   │        意義        │
  ├─────────────┼────────────────────┤
  │ 正值（> 0） │ 傾向「是這個人」   │
  ├─────────────┼────────────────────┤
  │ 負值（< 0） │ 傾向「不是這個人」 │
  ├─────────────┼────────────────────┤
  │ 越大越正    │ 越像               │
  └─────────────┴────────────────────┘

  然後 sigmoid(Score) 轉換到 0~1：
  - Score = 0 → Conf = 0.50
  - Score = 1 → Conf = 0.73
  - Score = -1 → Conf = 0.27

  預設閾值 SVM_UNKNOWN_THRESH = 0.60，等於 Score 需要 > 0.405 才算認識。

  單人模式（只有 Joey）
  Score 是 cosine 相似度，範圍 -1 ~ 1，負值同樣代表「方向相反、完全不像」。

  所以你在 console 看到的 [SVM] Scores=[Joey:1.23  Golden:-0.87] 這種輸出，Joey 正值大 = 選 Joey，Golden 負值 =
  確定不是 Golden。
  
## === 101: 側臉特徵的個人區分度比正臉低
❯ [SVM/右上] Scores=[henry:-0.79  Kirk:-0.80  Joey:-0.51]  獲勝=Joey sigmoid=0.376(閾0.60)  → Unknown,
  右上有36張圖,但查不出是Joey. 這樣還是樣本數不夠多嗎?

● 樣本數不是主因。注意看這組分數：

  henry:-0.79  Kirk:-0.80  Joey:-0.51

  三個人的分數全都是負數，代表這個特徵向量在 SVM 的決策空間裡，落在「所有人的邊界外面」。即使 Joey
  是最高分，sigmoid(-0.51) = 0.376，閾值 0.40 才能認出，比正臉要低很多。

  ---
  真正的原因：側臉特徵的個人區分度比正臉低

  正臉時，325 維的向量夾角特徵能清楚區分不同人的臉型比例。但臉轉到右上時：

  - MediaPipe 有部分點是估算出來的（非真實偵測）
  - 臉部向量夾角在大幅轉動後，人與人之間的差異被壓縮，三個人的特徵向量彼此更相近
  - 結果 OvR SVM 分數整體下移，所有人都偏負值
  
## === 102:
  設計文件說「夾角是幾何不變量，不受轉頭影響」——這個說法有條件限制

  這個說法成立的前提是：landmark 座標必須在頭部固定的 3D 座標系裡。

  但 MediaPipe 回傳的是攝影機影像座標系（x,y 對應畫面位置，z 是相對深度）。

  ---
  轉頭時發生什麼事

  正臉時                    臉轉右時
  左眼 ←───── 鼻尖           左眼 ← 鼻尖（透視壓縮）
  右眼 ─────→ 鼻尖               右眼 ──────────→ 鼻尖

  → v_左眼 和 v_右眼 的方向都變了
  → arccos(unit_左眼 · unit_右眼) 也跟著變

  透視效果（Perspective Foreshortening）讓近側臉部「壓縮」、遠側「拉伸」，導致所有向量夾角都隨轉頭而改變。

  ---
  所以這就是問題根源

  - 正臉 SVM 學到的是「正臉角度分布」
  - 右上 SVM 學到的是「右上角度分布」
  - 但若現在的右上角度跟訓練時的右上角度稍有不同，特徵向量就不一樣 → 分數掉

  ---
  真正的解法是：先把 landmark 轉回正臉座標系再取特徵

  取得 Landmarks3D（攝影機空間）
      │
      ▼
  估算頭部旋轉矩陣 R（MediaPipe 可算出 Yaw/Pitch/Roll）
      │
      ▼
  套用 R⁻¹ 把所有點「轉回正臉」
      │
      ▼
  對正規化後的 landmark 取 325 維夾角特徵
      │
      ▼
  單一 SVM 就夠了（特徵不再隨姿態變化）

  這樣做之後，各角度的同一個人特徵向量會接近，一個 SVM 就能處理所有姿態。

  要我幫你研究 MediaPipe 能不能直接拿到旋轉矩陣，或用 Yaw/Pitch 自己建 R⁻¹ 嗎？
  
## === 102-2: 
Q: 我想使用Mediapipe蒐集的人臉特徵,並用SVM來做人臉辨識,目前在處理細節上有看到二種說法.
1. 給 SVM 「向量的長度與夾角」,計算點對點的向量夾角： 比如「嘴角向量」與「下巴向量」的夾角。夾角是絕對不會因為轉頭而改變的（幾何不變量）。

2. 夾角是幾何不變量，不受轉頭影響. 這個說法成立的前提是：landmark 座標必須在頭部固定的 3D 座標系裡。臉轉右時, 左眼 ← 鼻尖（透視壓縮）,透視效果（Perspective Foreshortening）讓近側臉部「壓縮」、遠側「拉伸」，導致所有向量夾角都隨轉頭而改變。

這二種講法那個是對的?

A: 這是一個非常專業且深入的問題。直接給出結論：第二種說法才是實務上正確的。
第一種說法在「純幾何」的理想情況下成立，但在「電腦視覺（Camera Projection）」的現實中是錯誤的。
以下是詳細的分析與校正：
1. 為什麼第一種說法（夾角不變）在現實中會失效？
第一種說法忽略了**投影（Projection）**的過程。

3D vs 2D： 臉部是一個 3D 結構，但鏡頭捕捉到的是 2D 影像。當臉部旋轉時，雖然特徵點在 3D 空間的相對位置沒變，但投影到 2D 平面上的**「表觀夾角」**會因為透視投影（Perspective Projection）而產生扭曲。

透視壓縮（Perspective Foreshortening）： 正如第二種說法提到的，當臉轉向側面，靠近鏡頭的那半邊臉會看起來比較大，遠離的會縮小。這會導致原本在 3D 空間相等的角度，在 2D 畫面上變得完全不同。

2. MediaPipe 的特殊性：3D 還是 2D？
這點是你實作時最需要注意的細節。MediaPipe Face Mesh 會回傳兩種座標：

Image Coordinates (2D/Normalized): 直接對應圖片像素，這絕對會受轉頭影響。

Metric Landmark Coordinates (3D): MediaPipe 試圖預測一個 3D 空間座標（以公尺或公分為單位）。

但是！ 即使你用 MediaPipe 的 3D 座標，它依然是從 2D 影像「推估」出來的。當轉頭角度過大時，深度（Z 軸）的推估誤差會劇增，導致你計算出的 3D 夾角依然不穩定。

## === 106:
❯ 還有一個老問題,就是一個未訓練的人在detect時,很容易會辨識到另一個人. ex: [SVM-1P/全角度] 最大cosine=0.637
  sigmoid=0.654(閾0.60)  → RedBean. 但我是Joey. 除了修改閾值外,還有沒有其他的建議?

● 這是「開放集合辨識」(Open-Set Recognition) 的經典問題。除了調閾值，有幾個方向：

  ---
  問題根源（單人模式）

  目前只取最大 cosine 值：
  Sims = W @ xn   # 對所有訓練向量的 cosine 相似度
  MaxSim = max(Sims)  # 只看最高的一個
  200 個訓練向量裡只要有 1 個碰巧跟陌生人的臉相近，就會超過閾值。

  ---
  建議：改用 Top-K 平均取代最大值

  # 取相似度最高的前 10% 向量，取平均
  TopK    = max(1, int(len(Sims) * 0.10))
  TopKAvg = float(np.mean(np.sort(Sims)[-TopK:]))
  Conf    = sigmoid(TopKAvg)

  效果：陌生人偶爾有 1~2 個訓練向量相近，但前 10% 平均仍然低；真正的本人，前 10% 平均會明顯高。

  ---
  三種方案比較

  ┌───────────────────────┬────────┬────────────────────────┐
  │         方案          │  難度  │          效果          │
  ├───────────────────────┼────────┼────────────────────────┤
  │ 提高閾值（現有）      │ 最簡單 │ 治標，降低本人辨識率   │
  ├───────────────────────┼────────┼────────────────────────┤
  │ Top-K 平均            │ 改幾行 │ 對偶爾相似的陌生人有效 │
  ├───────────────────────┼────────┼────────────────────────┤
  │ One-Class SVM（SVDD） │ 複雜   │ 最嚴謹，需額外實作     │
  └───────────────────────┴────────┴────────────────────────┘

  ---

## === 107: ❯ COSINE_VERIFY_THRESH(餘弦驗證) 好像也會把原本正確的class,因為此值設太高,被filter掉. 還有什麼好建議?
● 目前 COSINE_VERIFY_THRESH
  的問題是它用「單一平均向量」做判斷，但每個人的訓練向量本身有分布，若閾值設太高，正確的人也會被 filter。

  更好的方法是 SVM 分差檢查（margin check）：看 top-1 和 top-2
  的原始分差。已知者的分差大（模型確定），陌生人的分差小（模型搖擺不定）。這個方法不需要 per-class
  平均向量，也不受訓練樣本分布影響。
  

  ---
  新增的 Unknown 偵測機制：分差閾值（Margin Threshold）

  原理：
  - 對已知的人：SVM 對他的那個分類器分數很高（例如 +2.0），其他人的 SVM 分數很低（例如 -0.8）→ 分差 = 2.8（大）
  - 對陌生人：所有 SVM 都沒有把握，分數都在 0 附近（例如 +0.3 / +0.1）→ 分差 = 0.2（小）

  這個 "margin" 是模型自己的不確定性指標，不需要 per-class 平均向量，不受訓練樣本分布影響。

  控制方式：
  - UI 新增了「分差閾值(Margin)」Slider，範圍 0.0 ~ 3.0，預設 0.50
  - 拖高 → 更嚴格（Unknown 更多）；拖低 → 更寬鬆
  - Console 輸出現在會印出 margin=X.XX(閾Y.YY) 方便觀察實際值來調整

  建議操作流程：
  1. 先用 Margin = 0.0（關閉），observe 一個已知人的 margin 值範圍（通常 1.0 ~ 3.0）
  2. 再測陌生人的 margin 值（通常 0.1 ~ 0.8）
  3. 在兩個範圍之間選一個切分點設為閾值
