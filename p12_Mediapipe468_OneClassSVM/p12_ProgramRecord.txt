● Step 2：把位移向量單位化（只保留方向）

  每個關鍵點到鼻尖的向量 v_i 除以自己的長度，變成長度為 1 的方向向量：

  v_i = [0.3, 0.5, 0.1]  → unit_i = [0.49, 0.82, 0.16]（長度=1）

  目的：消除臉離鏡頭距離的影響。靠近時向量變長、遠離時變短，但方向不變，單位化後只剩「方向」資訊。

  ---
  Step 3：計算 25 個方向向量兩兩之間的夾角（300 個）

  θ_ij = arccos(unit_i · unit_j)

  為什麼要算夾角？

  想像從鼻尖往左眼外角射一條線、再往右眼外角射一條線，這兩條線之間的夾角——不同人的臉型不同，這個夾角就不同。

  夾角的三個優點：

  ┌──────────────┬────────────────────────────────────────────────────────────┐
  │     特性     │                            說明                            │
  ├──────────────┼────────────────────────────────────────────────────────────┤
  │ 尺度不變     │ 臉大臉小、距離遠近，夾角不變                               │
  ├──────────────┼────────────────────────────────────────────────────────────┤
  │ 姿態不變     │ Step 0 已轉正，夾角在正臉座標系下計算                      │
  ├──────────────┼────────────────────────────────────────────────────────────┤
  │ 個人區分力強 │ 25 個點兩兩組合出 300 個角度，細膩描述每個人獨特的臉部幾何 │
  └──────────────┴────────────────────────────────────────────────────────────┘

  ---
  Part A + Part B 合在一起的意義：

  - Part A（300 個夾角）：臉部各部位的相對方向關係（臉型輪廓）
  - Part B（25 個正規化距離）：各部位離鼻尖的比例遠近（臉部比例）

  兩者合計 325 維，從不同角度描述同一張臉的幾何特徵。

.簡單來說,是否就是算二個三維空間的的距離?
● 不完全是，讓我說清楚：

  計算的不是「兩點之間的距離」，而是「單一向量的長度」（模）：

  v_i = [0.3, 0.5, 0.1]
  長度 = √(0.3² + 0.5² + 0.1²) = √(0.09 + 0.25 + 0.01) = √0.35 ≈ 0.592

  unit_i = v_i / 0.592 = [0.49, 0.82, 0.16]

  這個長度是**「關鍵點 i」到「鼻尖」的距離**（三維空間），只是一個點到原點的距離，不是兩個關鍵點之間的距離。

  ---
  整體流程的三種「距離/方向」概念：

  ┌────────┬───────────────────────────────────┬─────────┐
  │  步驟  │             計算的是              │  維度   │
  ├────────┼───────────────────────────────────┼─────────┤
  │ Step 1 │ 每個關鍵點到鼻尖的位移向量        │ (25, 3) │
  ├────────┼───────────────────────────────────┼─────────┤
  │ Step 2 │ 把這個向量除以自己的長度 → 純方向 │ (25, 3) │
  ├────────┼───────────────────────────────────┼─────────┤
  │ Part B │ 這個長度除以臉寬 → 比例距離       │ (25,)   │
  ├────────┼───────────────────────────────────┼─────────┤
  │ Step 3 │ 25 個方向向量兩兩夾角             │ (300,)  │
  └────────┴───────────────────────────────────┴─────────┘

  Step 3 的夾角才涉及「兩個點的關係」，但算的是方向之間的角度，不是座標間的距離。

● 是 SGDClassifier 的概念，但完全手寫的版本。

  _trainOneBinary() 的實作：

  # svm_classifier_np.py
  for _ in range(self._NEpochs):          # 200 epochs
      Indices = np.random.permutation(N)
      for Start in range(0, N, BatchSize): # mini-batch SGD
          Xb, Yb = X[Batch], Y[Batch]
          Scores = Xb @ w + b
          Margin = Yb * Scores
          Mask   = Margin < 1.0           # hinge loss 條件

          dw = -mean(Yb[Mask] * Xb[Mask]) + λ * w   # L2 正則化
          w -= lr * dw
          b -= lr * db

      lr *= LrDecay                       # 學習率衰減

❯ 主管擔心你寫的svm會和sklearn的svm有落差,你覺得這個的可能性有多高?

● 老實說，有落差的可能性偏高（我估 60~70%）。

  ---
  主要差距來源：

  1. 最大風險：收斂品質

  我們用的是手寫 mini-batch SGD，sklearn 的 LinearSVC 底層用的是
  liblinear（座標下降法），數學上保證收斂到全域最優解。我們的 SGD 只是近似解，200 epochs 不一定夠。

  2. 學習率調整

  我們用固定指數衰減（LrDecay=0.99），sklearn 有基於理論推導的最佳學習率排程，收斂更穩。

  3. 正則化施加方式

  我們的 L2 正則化是每個 mini-batch 手動加，sklearn 有更精確的實作。

  ---
  落差會表現在哪裡：

  ┌──────────────────┬────────────────────────────────┐
  │       現象       │              原因              │
  ├──────────────────┼────────────────────────────────┤
  │ SVM 分數偏低     │ 決策邊界沒收斂到最優，邊距較小 │
  ├──────────────────┼────────────────────────────────┤
  │ Margin 不穩定    │ 同上                           │
  ├──────────────────┼────────────────────────────────┤
  │ 類似臉孔較難區分 │ 最優邊界沒找到                 │
  └──────────────────┴────────────────────────────────┘

  如果部署環境允許 sklearn，可以做個對照實驗——把 SvmClassifier.fit() 換成 sklearn LinearSVC，用同樣的訓練資料比較 SVM
   分數高低。工程量不大，結論很清楚。

  │ 速度       │ 大資料快                      │ 小/中資料快                  │
  ├────────────┼───────────────────────────────┼──────────────────────────────┤
  │ 結果穩定性 │ 每次訓練略有不同（隨機性）    │ 每次結果相同                 │
  ├────────────┼───────────────────────────────┼──────────────────────────────┤
  │ 適合場景   │ 資料量非常大（百萬筆）        │ 資料量中小（數千筆以下）     │
  └────────────┴───────────────────────────────┴──────────────────────────────┘

  ---
  我們的場景：

  每人約 30~100 筆，總共幾百筆，這個資料量非常適合 LinearSVC——資料小、LinearSVC 速度快且解最優。

  SGDClassifier 的優勢（大資料）在我們這個場景完全用不到，反而帶來「不保證收斂」的缺點。

  ---
  結論：

  我一開始選擇手寫 SGD 是為了避免 sklearn 依賴。若要改用 sklearn，LinearSVC 比 SGDClassifier
  更適合我們這個小資料量的人臉辨識場景，品質更有保障。
